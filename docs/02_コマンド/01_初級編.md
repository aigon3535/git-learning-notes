

## add
- 変更対象に棚上げする操作。ステージングと呼びます。

```
git add
```

## commit
- 変更を確定する操作。
- 感覚的に「しっかりめに保存する」に近く、commitした変更は履歴として残されます。つまり、ユーザは意図的に履歴を残したい時、commitする。
- コミット単位は、目的別に分けることが好ましいです。

```
git commit
```

## push
- ローカルからリモートに反映します。
- ローカルにあるパワポ資料を、共有フォルダにある資料に上書きする行為に近いです。

```
git push
```


## 一般的な業務との比較
あるパワポ資料が共有フォルダにあり、ローカルにコピーして編集する場合です。
- `git clone`：ローカルに何も状態で、共有フォルダのパワポをローカルにコピーする
- `git pull`：共有フォルダのパワポをローカルのパワポに上書きする。中でも差分だけ。
- `git commit`：履歴を残して保存する。
- `git push`：ローカルの資料を共有フォルダ上書きする。Gitではそのまま上書きされずに「本当に上書きするの？どういう内容を」を皆でチェックする工程を挟みます＝プルリクエスト

特徴は、**変更確定に複数の工程をはさめる**ことです（`add`と`commit`と`push`）。
ソースコードを書いている時、手元でいくつか変更しつつ試したい時があります。下書きを書いている状態です。さらに反映には2段階あることで、混入を避ける、かつ、レビュー工程を挟むことができます。
